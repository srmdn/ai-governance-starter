# .gitignore for AI-Assisted Projects

`.gitignore` is a day-zero file. Create it before the first commit — not after
you've already leaked a `.env` or pushed your editor's temp files.

This starter kit ships a ready-to-use `.gitignore` at the repo root. Copy it
into any new project as-is.

---

## What's covered and why

### OS files
```
.DS_Store, Thumbs.db, desktop.ini, ...
```
macOS and Windows leave metadata files everywhere. They have no place in any
repository and cause constant noise in git status.

### Editor / IDE
```
.vscode/, .idea/, *.swp, *.swo, ...
```
Editor config is personal. Committing it forces your settings on every
contributor. If you want to share VS Code settings, use a `.vscode/extensions.json`
committed intentionally — not the whole folder.

### AI tool sessions / memory
```
.claude/, .codex/, .cursor/, .aider*, .continue/
```
AI tools write session state, memory, and config into hidden directories.
None of this belongs in version control — it contains personal context,
machine-specific paths, and sometimes token or API data.

### Environment / secrets
```
.env, .env.*, *.pem, *.key, secrets/, credentials/
```
The most critical section. A committed `.env` with real credentials is a
security incident. The rule: **only `.env.example` is ever committed**, with
all variable names and no real values.

`!.env.example` is explicitly un-ignored so it stays trackable.

### Node / npm / Bun
```
node_modules/, .npm/, .yarn/, dist/, .cache/, ...
```
`node_modules` can be hundreds of MB and is always reproducible from
`package.json`. Build caches are machine-specific. Neither belongs in git.

### Build output
```
dist/, build/, out/, .next/, .nuxt/, .astro/, ...
```
Build output is derived from source. Committing it creates conflicts and makes
diffs unreadable. Generate it on the server or in CI.

### Language-specific
```
Python: __pycache__/, .venv/, *.pyc
Go: *.exe, *.out, vendor/ (if not vendoring intentionally)
Rust: target/
Ruby: .bundle/, vendor/bundle/
Java/Kotlin: *.class, *.jar, .gradle/, build/
```
Each language has its own class of generated/compiled artifacts. These are all
reproducible from source and should never be committed.

### Database files
```
*.db, *.db-shm, *.db-wal, *.sqlite, *.sqlite3, *.sql.gz
```
SQLite database files are binary, conflict-prone, and often contain real user
data. Never commit them. If you need to share a schema, commit a migration
file — not the database itself.

`.db-shm` and `.db-wal` are SQLite write-ahead log files that appear during
active connections. They must also be ignored.

### Logs and temp files
```
*.log, logs/, *.tmp, *.bak, *.orig, tmp/, temp/
```
Logs grow unbounded and contain runtime data. Temp and backup files are
editor/tool artifacts. None of these are source.

### Coverage reports
```
coverage/, .coverage, *.lcov, lcov.info
```
Coverage is generated by your test runner. It's useful locally and in CI
dashboards, but has no business in git history.

## How to use

Copy the `.gitignore` from this repo into your project root before the first
commit:

```sh
cp /path/to/ai-governance-starter/.gitignore /path/to/your-project/
```

Then add it in your first commit alongside `CLAUDE.md` / `AGENTS.md`,
`.env.example`, and your governance files.

---

## Already committed something you shouldn't have?

If you've already committed a file that should be ignored:

```sh
# Remove from git tracking without deleting the file
git rm --cached path/to/file

# Then add it to .gitignore and commit
echo "path/to/file" >> .gitignore
git add .gitignore
git commit -m "stop tracking accidental file"
```

For secrets that were committed, treat it as a full credential rotation — the
secret is now in git history and must be considered compromised, even if you
remove it in a later commit. See `docs/AI-COMMIT-CLEANUP.md` for cleanup steps.
